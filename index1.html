TP 0: Mise en place d'un Cluster Kubernetes v1.28 avec Ansible

Objectifs du TP

Déployez automatiquement un cluster Kubernetes v1.28 complet (1 master + workers) à l'aide d'un rôle Ansible. Vous apprendrez à configurer l'inventaire, personnaliser les variables, exécuter le playbook et vérifier le bon fonctionnement du cluster.

Durée estimée : 90 minutes | Difficulté : Intermédiaire

Ce TP nécessite des machines Linux accessibles via SSH et Ansible installé sur la machine de contrôle.

Prérequis

Avant de commencer, assurez-vous d'avoir les éléments suivants :

  • Machines Linux : Ubuntu/Debian (recommandé) ou CentOS — au minimum 2 machines (1 master + 1 worker)
  • Accès SSH : L'utilisateur de contrôle doit pouvoir se connecter en SSH sans mot de passe (clés SSH configurées)
  • Ansible installé sur la machine de contrôle (version 2.9+)
  • Git installé sur la machine de contrôle
  • Droits sudo sur les machines cibles (master et worker)
  • Connectivité réseau entre toutes les machines et accès à Internet
Configuration recommandée par machine

CPU : 2 vCPUs minimum | RAM : 2 Go minimum | Disque : 20 Go | OS : Ubuntu 22.04 LTS

Vérification des prérequis
$ ansible --version
ansible [core 2.15.x]
$ git --version
git version 2.x.x
$ ssh user@172.16.201.22 "hostname"
master-node

Étape 2 — Structure du rôle Ansible

Explorons l'arborescence du projet cloné. Le rôle est organisé en 3 sous-rôles qui s'occupent de configurer les différentes parties du cluster :

STRUCTURE 
ansible-k8as-v1.28/
├── ansible.cfg          # Configuration Ansible (SSH, timeouts)
├── hosts.ini            # Inventaire des machines (master & workers)
├── main.yaml            # Playbook principal
└── roles/
    ├── common/          # Rôle commun (tous les nœuds)
    │   ├── defaults/
    │   │   └── main.yml     # Variables par défaut (versions K8s)
    │   ├── handlers/
    │   │   └── main.yml     # Handlers (restart containerd)
    │   ├── tasks/
    │   │   └── main.yml     # Tâches : dépendances, swap, containerd, kubeadm
    │   └── templates/
    │       └── containerd-config.toml.j2  # Template containerd
    ├── master/          # Rôle master
    │   ├── meta/
    │   │   └── main.yml     # Dépendance vers rôle common
    │   └── tasks/
    │       └── main.yml     # kubeadm init, kubeconfig, CNI Calico, join token
    └── worker/          # Rôle worker
        ├── meta/
        │   └── main.yml     # Dépendance vers rôle common
        └── tasks/
            └── main.yml     # Copie et exécution du join command
Rôle Description Actions principales
common Préparation de tous les nœuds Désactivation swap, installation containerd, kubeadm/kubelet/kubectl, configuration réseau (sysctl, modules kernel)
master Initialisation du nœud master kubeadm init, copie kubeconfig, installation CNI Calico, génération du join command
worker Jonction des nœuds workers Copie et exécution du join command généré par le master

Étape 3 — Configurer l'inventaire Ansible

L'inventaire définit les machines cibles du déploiement. Modifiez le fichier hosts.ini avec les adresses IP de vos machines :

INI 
# hosts.ini
[all:vars]
ansible_python_interpreter=/usr/bin/python3.11
ansible_user=adminefi

[master]
master-node ansible_host=172.16.201.22

[worker]
worker-node1 ansible_host=172.16.201.23
Personnalisation

Remplacez les adresses IP et le nom d'utilisateur (adminefi) par ceux de votre environnement. Vous pouvez ajouter autant de workers que nécessaire dans le groupe [worker].

Vérifiez que la connectivité Ansible fonctionne correctement :

Terminal
$ ansible all -i hosts.ini -m ping
master-node | SUCCESS => {
"changed": false,
"ping": "pong"
}
worker-node1 | SUCCESS => {
"changed": false,
"ping": "pong"
}

Étape 4 — Configurer les variables du rôle

Les variables par défaut du rôle se trouvent dans roles/common/defaults/main.yml. Vous pouvez les personnaliser selon vos besoins :

YAML 
# roles/common/defaults/main.yml
kubernetes_version: "1.28.0"
containerd_version: "1.6.9"

Examinons également le playbook principal main.yaml :

YAML 
# main.yaml
- hosts: master
  become: yes
  remote_user: adminefi
  roles:
    - master

- hosts: worker
  become: yes
  remote_user: adminefi
  roles:
    - worker
Fonctionnement du playbook

Le playbook exécute d'abord le rôle master (qui dépend de common) pour initialiser le nœud master, puis exécute le rôle worker (qui dépend aussi de common) pour joindre les workers au cluster. Le réseau CNI utilisé est Calico v3.26.1.

TP 16: Namespaces, ResourceQuota & LimitRange

Objectifs du TP

Apprenez à isoler les ressources entre équipes, limiter la consommation et garantir une allocation équitable des ressources du cluster.

Concepts

Qu'est-ce qu'un Namespace ?

Un Namespace est un mécanisme d'isolation logique dans Kubernetes. Il permet de :

  • Séparer les ressources entre différentes équipes ou projets
  • Appliquer des quotas de ressources par namespace
  • Définir des politiques réseau spécifiques
  • Gérer les accès RBAC de manière granulaire

Qu'est-ce qu'un ResourceQuota ?

Un ResourceQuota limite la quantité totale de ressources qu'un namespace peut consommer :

  • CPU et Mémoire : Limites sur requests et limits
  • Objets : Nombre de Pods, Services, ConfigMaps, etc.
  • Stockage : Espace disque total des PVC

Qu'est-ce qu'un LimitRange ?

Un LimitRange définit les contraintes par défaut et les limites pour les conteneurs individuels :

  • default : Valeurs par défaut si non spécifiées
  • defaultRequest : Requests par défaut
  • min/max : Limites minimales et maximales autorisées

Exercices Pratiques

Exercice 1 : Créer un Namespace

Créez un namespace pour isoler les ressources d'une équipe de développement.

YAML 
# namespace-dev.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: dev-team
  labels:
    environment: development
    team: backend
Terminal
$ kubectl apply -f namespace-dev.yaml
namespace/dev-team created
$ kubectl get namespaces
NAME STATUS AGE
default Active 10d
dev-team Active 5s
kube-system Active 10d

Exercice 2 : Appliquer un ResourceQuota

Limitez les ressources totales que le namespace peut utiliser.

YAML 
# resource-quota.yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: dev-quota
  namespace: dev-team
spec:
  hard:
    # Limites CPU
    requests.cpu: "4"
    limits.cpu: "8"
    # Limites Mémoire
    requests.memory: 4Gi
    limits.memory: 8Gi
    # Nombre d'objets
    pods: "20"
    services: "10"
    configmaps: "10"
    secrets: "10"
    persistentvolumeclaims: "5"
    # Stockage
    requests.storage: 20Gi
Terminal
$ kubectl apply -f resource-quota.yaml
resourcequota/dev-quota created
$ kubectl describe quota dev-quota -n dev-team
Name: dev-quota
Namespace: dev-team
Resource Used Hard
-------- ---- ----
configmaps 0 10
limits.cpu 0 8
limits.memory 0 8Gi
pods 0 20
requests.cpu 0 4
requests.memory 0 4Gi

Exercice 3 : Configurer un LimitRange

Définissez des limites par défaut pour chaque conteneur.

YAML 
# limit-range.yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: dev-limits
  namespace: dev-team
spec:
  limits:
  - type: Container
    default:
      cpu: "500m"
      memory: "256Mi"
    defaultRequest:
      cpu: "100m"
      memory: "128Mi"
    min:
      cpu: "50m"
      memory: "64Mi"
    max:
      cpu: "2"
      memory: "1Gi"
  - type: PersistentVolumeClaim
    min:
      storage: 1Gi
    max:
      storage: 10Gi
Terminal
$ kubectl apply -f limit-range.yaml
limitrange/dev-limits created
$ kubectl describe limitrange dev-limits -n dev-team
Name: dev-limits
Namespace: dev-team
Type Resource Min Max Default DefaultRequest
---- -------- --- --- ------- --------------
Container cpu 50m 2 500m 100m
Container memory 64Mi 1Gi 256Mi 128Mi

Exercice 5 : Tester le dépassement de Quota

Essayez de créer un Pod qui dépasse les limites maximales.

YAML 
# pod-exceed-limits.yaml
apiVersion: v1
kind: Pod
metadata:
  name: greedy-pod
  namespace: dev-team
spec:
  containers:
  - name: nginx
    image: nginx:alpine
    resources:
      requests:
        cpu: "4"          # Dépasse max du LimitRange
        memory: "2Gi"      # Dépasse max du LimitRange
Terminal
$ kubectl apply -f pod-exceed-limits.yaml
Error from server (Forbidden): error when creating "pod-exceed-limits.yaml":
pods "greedy-pod" is forbidden: [maximum cpu usage per Container is 2, but limit is 4,
maximum memory usage per Container is 1Gi, but limit is 2Gi]
Attention

Le LimitRange empêche la création de Pods qui dépassent les limites maximales définies.

TP 17: API Gateway dans Kubernetes — Concepts, Ingress vs Gateway & Mise en Pratique

Objectifs du TP

Comprendre les différences entre un Ingress Controller et une API Gateway, déployer deux applications de test, les exposer via Nginx Ingress puis via Kong API Gateway avec rate-limiting, et comparer les deux approches en production.

Durée estimée : 90 minutes | Difficulté : Avancé

Ce TP nécessite un cluster Kubernetes fonctionnel avec kubectl et helm installés.

1. Concepts théoriques

Qu'est-ce qu'un Ingress ?

Un Ingress dans Kubernetes est une ressource qui gère l'accès externe aux services du cluster, principalement via HTTP/HTTPS. Il fonctionne grâce à deux composants :

  • Ingress Resource : un objet Kubernetes qui définit les règles de routage (host, path → service)
  • Ingress Controller : le composant qui implémente réellement ces règles (Nginx, Traefik, HAProxy, etc.)

Limites de l'Ingress :

  • Pas de gestion native de l'authentification avancée (OAuth2, JWT, mTLS)
  • Pas de rate limiting intégré
  • Pas de transformation de requêtes/réponses
  • Pas de circuit breaker natif
  • Observabilité limitée aux logs et métriques basiques

Qu'est-ce qu'une API Gateway ?

Une API Gateway est un point d'entrée centralisé et intelligent pour les APIs. Elle offre des fonctionnalités bien au-delà du simple routage :

  • Authentification & Autorisation : OAuth2, JWT, API Keys, mTLS
  • Rate Limiting : contrôle du débit des requêtes par client/route
  • Circuit Breaker : protection contre les services défaillants
  • Transformation : modification des requêtes/réponses en transit
  • Observabilité : métriques détaillées, tracing distribué, logging structuré
API Gateways dans l'écosystème Kubernetes

Kong, Traefik Hub, APISIX, AWS API Gateway, KrakenD, Istio Gateway, Envoy Gateway

Tableau comparatif : Ingress vs API Gateway

Critère Ingress (Nginx) API Gateway (Kong)
Routage HTTP/HTTPS ✅ Natif (host, path) ✅ Avancé (regex, headers, méthodes)
Authentification & Autorisation ⚠️ Basic Auth, External Auth ✅ OAuth2, JWT, API Keys, mTLS, LDAP
Rate Limiting ⚠️ Annotations limitées ✅ Plugin natif (par route, consumer, IP)
Transformation requêtes/réponses ❌ Non supporté ✅ Plugins Request/Response Transformer
Circuit Breaker ❌ Non supporté ✅ Plugin natif
Observabilité avancée ⚠️ Métriques/logs basiques ✅ Prometheus, Datadog, OpenTelemetry
Complexité de mise en place ✅ Simple et rapide ⚠️ Plus complexe, plus de configuration
Cas d'usage recommandé Sites web, apps simples APIs REST/GraphQL, microservices complexes

2. Schéma d'architecture

Le flux de trafic diffère selon l'approche choisie :

ARCHITECTURE

│                        APPROCHE INGRESS                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │

TP 1: Pods - L'unité de base

Objectifs du TP

Comprendre et manipuler les Pods, l'unité de déploiement la plus petite dans Kubernetes.

Concepts

Qu'est-ce qu'un Pod ?

Un Pod est l'unité de base dans Kubernetes. Il représente un ou plusieurs conteneurs qui :

  • Partagent le même espace réseau (même IP)
  • Peuvent partager des volumes de stockage
  • Sont toujours co-localisés sur le même nœud
  • Sont démarrés et arrêtés ensemble

Cycle de vie d'un Pod

Phase Description
Pending Le Pod est accepté mais pas encore planifié
Running Le Pod est lié à un nœud et les conteneurs démarrent
Succeeded Tous les conteneurs ont terminé avec succès
Failed Au moins un conteneur a échoué
Unknown État impossible à déterminer

TP 2: ReplicaSet - Haute Disponibilité

Objectifs du TP

Garantir la disponibilité de vos applications en maintenant automatiquement le nombre de répliques souhaité.

Concepts

Qu'est-ce qu'un ReplicaSet ?

Un ReplicaSet garantit qu'un nombre spécifié de répliques de Pod sont en cours d'exécution à tout moment.

  • Haute disponibilité : Si un Pod tombe, il est automatiquement recréé
  • Scaling : Augmentez ou diminuez le nombre de répliques
  • Sélecteurs : Identifie les Pods à gérer via des labels
Note importante

En pratique, on utilise rarement les ReplicaSets directement. On préfère les Deployments qui gèrent les ReplicaSets pour nous et offrent des fonctionnalités supplémentaires.

Exercices Pratiques

Exercice 1 : Créer un ReplicaSet

YAML 
# replicaset-nginx.yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx-rs
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.24
        ports:
        - containerPort: 80

TP 3: Deployment - Déploiement Déclaratif

Objectifs du TP

Maîtriser les Deployments : rolling updates, rollbacks, et stratégies de déploiement.

Concepts

Qu'est-ce qu'un Deployment ?

Un Deployment fournit des mises à jour déclaratives pour les Pods et les ReplicaSets.

  • Rolling Updates : Mises à jour progressives sans interruption
  • Rollback : Retour à une version précédente
  • Historique : Conservation des révisions
  • Pause/Resume : Contrôle fin des déploiements

Exercices Pratiques

Exercice 1 : Créer un Deployment

# deployment-nginx.yaml
apiVersion: apps/v1

  name: nginx-deploy

    app: nginx

  replicas: 3

    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.23
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: "100m"
            memory: "128Mi"
          limits:
            cpu: "200m"
            memory: "256Mi"

Exercice 5 : Stratégie de déploiement

YAML 
# deployment-strategy.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-rolling
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1           # Max pods au-dessus du nombre désiré
      maxUnavailable: 1    # Max pods indisponibles
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: app
        image: nginx:alpine

YAML

apiVersion: apps/v1

    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: app
        image: nginx:alpine

La stratégie Recreate cause une interruption de service ! Utilisez-la uniquement quand c'est nécessaire (incompatibilité de versions, etc.).

Objectifs

Étape 3 — Créer un Ingress

Créez une ressource Ingress pour router le trafic vers vos services :

YAML

apiVersion: networking.k8s.io/v1

spec:

  rules:

    http:
      paths:
      - pathType: Prefix
        path: "/"
        backend:
          service:
            name: svc-nginxs
            port:
              number: 80

TP 15: HPA - Horizontal Pod Autoscaler

Objectifs

Configurer l'autoscaling horizontal basé sur la charge CPU/mémoire pour adapter automatiquement le nombre de répliques.

Prérequis : Metrics Server

Terminal
$ minikube addons enable metrics-server

NAME CPU(cores) MEMORY(bytes)
nginx-deploy-6b8d4f9b7c-xxx 1m 5Mi

Exercice 1 : Créer un HPA

YAML 
# hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nginx-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nginx-deploy
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
