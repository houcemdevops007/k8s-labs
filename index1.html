TP 16: Namespaces, ResourceQuota & LimitRange

Objectifs du TP

Apprenez à isoler les ressources entre équipes, limiter la consommation et garantir une allocation équitable des ressources du cluster.

Concepts

Qu'est-ce qu'un Namespace ?

Un Namespace est un mécanisme d'isolation logique dans Kubernetes. Il permet de :

  • Séparer les ressources entre différentes équipes ou projets
  • Appliquer des quotas de ressources par namespace
  • Définir des politiques réseau spécifiques
  • Gérer les accès RBAC de manière granulaire

Qu'est-ce qu'un ResourceQuota ?

Un ResourceQuota limite la quantité totale de ressources qu'un namespace peut consommer :

  • CPU et Mémoire : Limites sur requests et limits
  • Objets : Nombre de Pods, Services, ConfigMaps, etc.
  • Stockage : Espace disque total des PVC

Qu'est-ce qu'un LimitRange ?

Un LimitRange définit les contraintes par défaut et les limites pour les conteneurs individuels :

  • default : Valeurs par défaut si non spécifiées
  • defaultRequest : Requests par défaut
  • min/max : Limites minimales et maximales autorisées

Exercices Pratiques

Exercice 1 : Créer un Namespace

Créez un namespace pour isoler les ressources d'une équipe de développement.

YAML 
# namespace-dev.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: dev-team
  labels:
    environment: development
    team: backend
Terminal
$ kubectl apply -f namespace-dev.yaml
namespace/dev-team created
$ kubectl get namespaces
NAME STATUS AGE
default Active 10d
dev-team Active 5s
kube-system Active 10d

Exercice 2 : Appliquer un ResourceQuota

Limitez les ressources totales que le namespace peut utiliser.

YAML 
# resource-quota.yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: dev-quota
  namespace: dev-team
spec:
  hard:
    # Limites CPU
    requests.cpu: "4"
    limits.cpu: "8"
    # Limites Mémoire
    requests.memory: 4Gi
    limits.memory: 8Gi
    # Nombre d'objets
    pods: "20"
    services: "10"
    configmaps: "10"
    secrets: "10"
    persistentvolumeclaims: "5"
    # Stockage
    requests.storage: 20Gi
Terminal
$ kubectl apply -f resource-quota.yaml
resourcequota/dev-quota created
$ kubectl describe quota dev-quota -n dev-team
Name: dev-quota
Namespace: dev-team
Resource Used Hard
-------- ---- ----
configmaps 0 10
limits.cpu 0 8
limits.memory 0 8Gi
pods 0 20
requests.cpu 0 4
requests.memory 0 4Gi

Exercice 3 : Configurer un LimitRange

Définissez des limites par défaut pour chaque conteneur.

YAML 
# limit-range.yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: dev-limits
  namespace: dev-team
spec:
  limits:
  - type: Container
    default:
      cpu: "500m"
      memory: "256Mi"
    defaultRequest:
      cpu: "100m"
      memory: "128Mi"
    min:
      cpu: "50m"
      memory: "64Mi"
    max:
      cpu: "2"
      memory: "1Gi"
  - type: PersistentVolumeClaim
    min:
      storage: 1Gi
    max:
      storage: 10Gi
Terminal
$ kubectl apply -f limit-range.yaml
limitrange/dev-limits created
$ kubectl describe limitrange dev-limits -n dev-team
Name: dev-limits
Namespace: dev-team
Type Resource Min Max Default DefaultRequest
---- -------- --- --- ------- --------------
Container cpu 50m 2 500m 100m
Container memory 64Mi 1Gi 256Mi 128Mi

Exercice 4 : Tester les Quotas

Déployez un Pod et observez l'application automatique des limites.

YAML 
# test-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: test-pod
  namespace: dev-team
spec:
  containers:
  - name: nginx
    image: nginx:alpine
    # Sans spécifier resources, LimitRange applique les défauts
Terminal
$ kubectl apply -f test-pod.yaml
pod/test-pod created
$ kubectl get pod test-pod -n dev-team -o yaml | grep -A 10 resources
resources:
limits:
cpu: 500m
memory: 256Mi
requests:
cpu: 100m
memory: 128Mi
Résultat

Le LimitRange a automatiquement ajouté les requests et limits par défaut au Pod !

Exercice 5 : Tester le dépassement de Quota

Essayez de créer un Pod qui dépasse les limites maximales.

YAML 
# pod-exceed-limits.yaml
apiVersion: v1
kind: Pod
metadata:
  name: greedy-pod
  namespace: dev-team
spec:
  containers:
  - name: nginx
    image: nginx:alpine
    resources:
      requests:
        cpu: "4"          # Dépasse max du LimitRange
        memory: "2Gi"      # Dépasse max du LimitRange
Terminal
$ kubectl apply -f pod-exceed-limits.yaml
Error from server (Forbidden): error when creating "pod-exceed-limits.yaml":
pods "greedy-pod" is forbidden: [maximum cpu usage per Container is 2, but limit is 4,
maximum memory usage per Container is 1Gi, but limit is 2Gi]
Attention

Le LimitRange empêche la création de Pods qui dépassent les limites maximales définies.

TP 1: Pods - L'unité de base

Objectifs du TP

Comprendre et manipuler les Pods, l'unité de déploiement la plus petite dans Kubernetes.

Concepts

Qu'est-ce qu'un Pod ?

Un Pod est l'unité de base dans Kubernetes. Il représente un ou plusieurs conteneurs qui :

  • Partagent le même espace réseau (même IP)
  • Peuvent partager des volumes de stockage
  • Sont toujours co-localisés sur le même nœud
  • Sont démarrés et arrêtés ensemble

Cycle de vie d'un Pod

Phase Description
Pending Le Pod est accepté mais pas encore planifié
Running Le Pod est lié à un nœud et les conteneurs démarrent
Succeeded Tous les conteneurs ont terminé avec succès
Failed Au moins un conteneur a échoué
Unknown État impossible à déterminer

Exercices Pratiques

Exercice 1 : Créer un Pod simple

Créez votre premier Pod avec un conteneur nginx.

YAML 
# mon-premier-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: mon-premier-pod
  labels:
    app: nginx
    env: dev
spec:
  containers:
  - name: nginx
    image: nginx:1.24
    ports:
    - containerPort: 80
Terminal
$ kubectl apply -f mon-premier-pod.yaml
pod/mon-premier-pod created
$ kubectl get pods
NAME READY STATUS RESTARTS AGE
mon-premier-pod 1/1 Running 0 10s

Exercice 2 : Inspecter un Pod

Utilisez différentes commandes pour examiner votre Pod.

Terminal
$ kubectl describe pod mon-premier-pod
Name: mon-premier-pod
Namespace: default
Priority: 0
Node: minikube/192.168.49.2
...
$ kubectl logs mon-premier-pod
/docker-entrypoint.sh: Configuration complete; ready for start up
$ kubectl get pod mon-premier-pod -o yaml

Exercice 3 : Exécuter des commandes dans un Pod

Accédez au shell du conteneur pour explorer.

Terminal
$ kubectl exec -it mon-premier-pod -- /bin/bash
root@mon-premier-pod:/# hostname
mon-premier-pod
root@mon-premier-pod:/# curl localhost
<!DOCTYPE html>...

Exercice 4 : Pod multi-conteneurs

Créez un Pod avec deux conteneurs qui partagent un volume.

YAML 
# pod-multi-conteneurs.yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-sidecar
spec:
  containers:
  - name: app
    image: nginx:alpine
    volumeMounts:
    - name: logs
      mountPath: /var/log/nginx
  - name: sidecar
    image: busybox
    command: ["sh", "-c", "tail -f /logs/access.log"]
    volumeMounts:
    - name: logs
      mountPath: /logs
  volumes:
  - name: logs
    emptyDir: {}

Exercice 5 : Supprimer un Pod

Terminal

$ kubectl delete -f mon-premier-pod.yaml

TP 2: ReplicaSet - Haute Disponibilité

Objectifs du TP

Garantir la disponibilité de vos applications en maintenant automatiquement le nombre de répliques souhaité.

Concepts

Qu'est-ce qu'un ReplicaSet ?

Un ReplicaSet garantit qu'un nombre spécifié de répliques de Pod sont en cours d'exécution à tout moment.

  • Haute disponibilité : Si un Pod tombe, il est automatiquement recréé
  • Scaling : Augmentez ou diminuez le nombre de répliques
  • Sélecteurs : Identifie les Pods à gérer via des labels
Note importante

En pratique, on utilise rarement les ReplicaSets directement. On préfère les Deployments qui gèrent les ReplicaSets pour nous et offrent des fonctionnalités supplémentaires.

Exercices Pratiques

Exercice 1 : Créer un ReplicaSet

YAML 
# replicaset-nginx.yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx-rs
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.24
        ports:
        - containerPort: 80
Terminal
$ kubectl apply -f replicaset-nginx.yaml
replicaset.apps/nginx-rs created
$ kubectl get rs
NAME DESIRED CURRENT READY AGE
nginx-rs 3 3 3 10s
$ kubectl get pods
NAME READY STATUS RESTARTS AGE
nginx-rs-7h4xj 1/1 Running 0 15s
nginx-rs-9k2lm 1/1 Running 0 15s
nginx-rs-b3nqp 1/1 Running 0 15s

Exercice 2 : Tester l'auto-réparation

Supprimez un Pod et observez le ReplicaSet le recréer automatiquement.

Terminal
$ kubectl delete pod nginx-rs-7h4xj
pod "nginx-rs-7h4xj" deleted
$ kubectl get pods
NAME READY STATUS RESTARTS AGE
nginx-rs-9k2lm 1/1 Running 0 2m
nginx-rs-b3nqp 1/1 Running 0 2m
nginx-rs-x5cvd 1/1 Running 0 5s <-- Nouveau Pod !
Auto-réparation

Le ReplicaSet détecte qu'il n'y a plus que 2 Pods au lieu de 3 et en crée immédiatement un nouveau !

Exercice 3 : Scaling manuel

Terminal
$ kubectl scale rs nginx-rs --replicas=5
replicaset.apps/nginx-rs scaled
$ kubectl get pods
NAME READY STATUS RESTARTS AGE
nginx-rs-9k2lm 1/1 Running 0 5m
nginx-rs-b3nqp 1/1 Running 0 5m
nginx-rs-x5cvd 1/1 Running 0 3m
nginx-rs-m8hjk 1/1 Running 0 10s
nginx-rs-p2qrs 1/1 Running 0 10s

Exercice 4 : Réduire les répliques

Terminal
$ kubectl scale rs nginx-rs --replicas=2
replicaset.apps/nginx-rs scaled
$ kubectl get pods
NAME READY STATUS RESTARTS AGE
nginx-rs-9k2lm 1/1 Running 0 6m
nginx-rs-b3nqp 1/1 Running 0 6m
nginx-rs-x5cvd 1/1 Terminating 0 4m

Commandes Essentielles

Commande Description
kubectl get rs Lister les ReplicaSets
kubectl describe rs <nom> Détails d'un ReplicaSet
kubectl scale rs <nom> --replicas=N Changer le nombre de répliques
kubectl delete rs <nom> Supprimer (et ses Pods)
Checklist
Créer un ReplicaSet avec 3 répliques
Tester l'auto-réparation
Scale up à 5 répliques
Scale down à 2 répliques

TP 3: Deployment - Déploiement Déclaratif

Objectifs du TP

Maîtriser les Deployments : rolling updates, rollbacks, et stratégies de déploiement.

Concepts

Qu'est-ce qu'un Deployment ?

Un Deployment fournit des mises à jour déclaratives pour les Pods et les ReplicaSets.

  • Rolling Updates : Mises à jour progressives sans interruption
  • Rollback : Retour à une version précédente
  • Historique : Conservation des révisions
  • Pause/Resume : Contrôle fin des déploiements

Exercices Pratiques

Exercice 1 : Créer un Deployment

YAML 
# deployment-nginx.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploy
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.23
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: "100m"
            memory: "128Mi"
          limits:
            cpu: "200m"
            memory: "256Mi"
Terminal
$ kubectl apply -f deployment-nginx.yaml
deployment.apps/nginx-deploy created
$ kubectl get deployments
NAME READY UP-TO-DATE AVAILABLE AGE
nginx-deploy 3/3 3 3 15s
$ kubectl get rs
NAME DESIRED CURRENT READY AGE
nginx-deploy-6b8d4f9b7c 3 3 3 20s

Exercice 2 : Rolling Update

Mettez à jour l'image nginx de 1.23 vers 1.24.

Terminal
$ kubectl set image deployment/nginx-deploy nginx=nginx:1.24
deployment.apps/nginx-deploy image updated
$ kubectl rollout status deployment/nginx-deploy
Waiting for deployment "nginx-deploy" rollout to finish: 1 out of 3 new replicas...
Waiting for deployment "nginx-deploy" rollout to finish: 2 out of 3 new replicas...
deployment "nginx-deploy" successfully rolled out

Exercice 3 : Voir l'historique des révisions

Terminal
$ kubectl rollout history deployment/nginx-deploy
deployment.apps/nginx-deploy
REVISION CHANGE-CAUSE
1 <none>
2 <none>
$ kubectl rollout history deployment/nginx-deploy --revision=2

Exercice 4 : Rollback

Revenez à la version précédente.

Terminal
$ kubectl rollout undo deployment/nginx-deploy
deployment.apps/nginx-deploy rolled back
$ kubectl rollout undo deployment/nginx-deploy --to-revision=1
deployment.apps/nginx-deploy rolled back

Exercice 5 : Stratégie de déploiement

YAML 
# deployment-strategy.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-rolling
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1           # Max pods au-dessus du nombre désiré
      maxUnavailable: 1    # Max pods indisponibles
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: app
        image: nginx:alpine

Exercice 6 : Stratégie Recreate

YAML 
# deployment-recreate.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-recreate
spec:
  replicas: 3
  strategy:
    type: Recreate  # Supprime tous les pods avant d'en créer de nouveaux
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: app
        image: nginx:alpine
Attention

La stratégie Recreate cause une interruption de service ! Utilisez-la uniquement quand c'est nécessaire (incompatibilité de versions, etc.).

Commandes Essentielles

Commande Description
kubectl get deploy Lister les Deployments
kubectl rollout status deploy/<nom> Suivre le statut du déploiement
kubectl rollout history deploy/<nom> Voir l'historique
kubectl rollout undo deploy/<nom> Rollback
kubectl set image deploy/<nom> <container>=<image> Mettre à jour l'image
kubectl rollout pause deploy/<nom> Mettre en pause
kubectl rollout resume deploy/<nom> Reprendre
Checklist
Créer un Deployment
Effectuer un Rolling Update
Voir l'historique des révisions
Effectuer un Rollback
Configurer maxSurge et maxUnavailable
Tester la stratégie Recreate

TP 4: Services - Exposition des Applications

Objectifs du TP

Exposer vos applications en interne et en externe avec les différents types de Services Kubernetes.

Concepts

Types de Services

Type Description Usage
ClusterIP IP interne au cluster (défaut) Communication inter-pods
NodePort Port sur chaque nœud (30000-32767) Accès externe simple
LoadBalancer Load balancer cloud Production cloud
ExternalName Alias DNS vers service externe Intégration externe

Exercices Pratiques

Exercice 1 : Service ClusterIP

YAML 
# service-clusterip.yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: ClusterIP
  selector:
    app: nginx
  ports:
  - port: 80          # Port du service
    targetPort: 80    # Port du conteneur
    protocol: TCP
Terminal
$ kubectl apply -f service-clusterip.yaml
service/nginx-service created
$ kubectl get svc
NAME TYPE CLUSTER-IP EXTERNAL-IP PORT(S) AGE
nginx-service ClusterIP 10.96.45.123 <none> 80/TCP 10s

Exercice 2 : Tester le Service en interne

Terminal
$ kubectl run test-pod --image=busybox --rm -it -- wget -qO- nginx-service
<!DOCTYPE html>...
$ kubectl run test-pod --image=busybox --rm -it -- wget -qO- nginx-service.default.svc.cluster.local

Exercice 3 : Service NodePort

YAML 
# service-nodeport.yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-nodeport
spec:
  type: NodePort
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
    nodePort: 30080   # Port externe (30000-32767)
Terminal
$ kubectl apply -f service-nodeport.yaml
$ kubectl get svc nginx-nodeport
NAME TYPE CLUSTER-IP EXTERNAL-IP PORT(S) AGE
nginx-nodeport NodePort 10.96.78.234 <none> 80:30080/TCP 5s
$ curl http://<NODE_IP>:30080

Exercice 4 : Service LoadBalancer

YAML 
# service-loadbalancer.yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-lb
spec:
  type: LoadBalancer
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
Note

Le type LoadBalancer nécessite un cloud provider. Sur Minikube, utilisez minikube tunnel pour simuler un load balancer.

Exercice 5 : Créer rapidement avec expose

Terminal

service/nginx-quick exposed

TP 5: ConfigMap - Externaliser la Configuration

Objectifs du TP

Séparer la configuration du code applicatif pour une meilleure portabilité.

Exercices Pratiques

Exercice 1 : Créer un ConfigMap

YAML 
# configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_ENV: "production"
  LOG_LEVEL: "info"
  MAX_CONNECTIONS: "100"
  app.properties: |
    server.port=8080
    database.host=db.example.com
    database.port=5432
Terminal
$ kubectl apply -f configmap.yaml
configmap/app-config created
$ kubectl create configmap db-config --from-literal=DB_HOST=localhost --from-literal=DB_PORT=5432
$ kubectl create configmap file-config --from-file=config.json

Exercice 2 : Utiliser comme variables d'environnement

YAML

TP 10: Probes - Surveillance de Santé

Objectifs du TP

Configurer les health checks pour garantir la disponibilité et la fiabilité de vos applications.

Architecture des Probes

Types de Probes

TypeDescription
livenessProbeVérifie si le conteneur est vivant. Échec = redémarrage
readinessProbeVérifie si le conteneur est prêt à recevoir du trafic
startupProbePour les applications lentes à démarrer

Exercice : Configurer les Probes

YAML 
# pod-probes.yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-healthy
spec:
  containers:
  - name: app
    image: nginx:alpine
    ports:
    - containerPort: 80
    livenessProbe:
      httpGet:
        path: /
        port: 80
      initialDelaySeconds: 5
      periodSeconds: 10
      failureThreshold: 3
    readinessProbe:
      httpGet:
        path: /
        port: 80
      initialDelaySeconds: 3
      periodSeconds: 5
    startupProbe:
      httpGet:
        path: /
        port: 80
      failureThreshold: 30
      periodSeconds: 10

Types de vérification

  • httpGet : Requête HTTP GET
  • tcpSocket : Connexion TCP
  • exec : Commande dans le conteneur

TP 15: HPA - Horizontal Pod Autoscaler

Objectifs

Configurer l'autoscaling horizontal basé sur la charge CPU/mémoire pour adapter automatiquement le nombre de répliques.

Prérequis : Metrics Server

Terminal
$ minikube addons enable metrics-server
✅ The 'metrics-server' addon is enabled
$ kubectl top pods
NAME CPU(cores) MEMORY(bytes)
nginx-deploy-6b8d4f9b7c-xxx 1m 5Mi

Exercice 1 : Créer un HPA

YAML 
# hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nginx-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nginx-deploy
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
Terminal
$ kubectl autoscale deployment nginx-deploy --cpu-percent=50 --min=2 --max=10
horizontalpodautoscaler.autoscaling/nginx-deploy autoscaled
$ kubectl get hpa
NAME REFERENCE TARGETS MINPODS MAXPODS REPLICAS AGE
nginx-deploy Deployment/nginx-deploy 10%/50% 2 10 2 30s

Exercice 2 : Générer de la charge

Terminal
$ kubectl run load-generator --image=busybox --restart=Never -- \
/bin/sh -c "while true; do wget -q -O- http://nginx-service; done"
$ watch kubectl get hpa
NAME REFERENCE TARGETS MINPODS MAXPODS REPLICAS
nginx-deploy Deployment/nginx-deploy 78%/50% 2 10 4
Résultat

Le HPA a détecté la charge élevée et a augmenté le nombre de répliques de 2 à 4 automatiquement !
